## 1. Architecture Overview

The eMule web server is a self-contained HTTP/HTTPS server embedded in the main Windows process. It exposes the full eMule feature set (downloads, uploads, search, servers, Kademlia, preferences, statistics) through a browser-facing HTML interface. There is no structured (JSON/XML) API — all responses are HTML, generated by string-replacement into one of two `.tmpl` files.

```
┌──────────────────────────────────────────────────────────────┐
│  eMule main UI thread (MFC message loop)                     │
│                                                              │
│  Global singletons: theApp.downloadqueue, uploadqueue,       │
│  sharedfiles, serverlist, searchlist, clientlist, friendlist │
└────────────────────────┬─────────────────────────────────────┘
                         │  WM_* messages (SendMessage)
                         │  WEB_GUI_INTERACTION
                         │  WEB_ADDDOWNLOADS
                         │  WEB_CATPRIO
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  CWebServer (started via StartServer / StopServer)           │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  WebSocketListeningFunc (worker thread)              │   │
│  │  • binds to WSPort / WSBindAddr                      │   │
│  │  • accept() loop                                     │   │
│  │  • spawns WebSocketAcceptedFunc per connection       │   │
│  └──────────────────────┬───────────────────────────────┘   │
│                          │ one thread per HTTP request        │
│  ┌───────────────────────▼───────────────────────────────┐   │
│  │  WebSocketAcceptedFunc                                │   │
│  │  • WSAEventSelect (FD_READ / FD_WRITE / FD_CLOSE)     │   │
│  │  • receives HTTP request headers + body               │   │
│  │  • calls OnRequestReceived()                          │   │
│  │    → static file? _ProcessFileReq()                   │   │
│  │    → dynamic page? _ProcessURL()                      │   │
│  │  • sends response via chunk queue                     │   │
│  └───────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

**Key constraint:** `_ProcessURL()` reads `theApp.*` singletons directly from the per-request worker thread **without any synchronisation**. The source acknowledges this:

```cpp
// Here we are in real trouble! We are accessing the entire emule main thread
// data without any synchronization!! Either we use the message pump for
// m_pdlgEmule or use some hundreds of critical sections...
// For now, an exception handler should prevent the worst things.
```

## 3. Threading & Socket Model

### Listening thread

`WebSocketListeningFunc` runs for the lifetime of the web server. It creates a TCP socket, binds to `thePrefs.GetWSPort()` on `thePrefs.GetBindAddrA()` (or `INADDR_ANY`), and loops on `accept()`.

### Per-connection thread

Every accepted connection spawns `WebSocketAcceptedFunc`. This function:

1. Allocates a `CWebSocket` wrapper around the accepted `SOCKET`.
2. Creates a `WSAEventSelect` event mask for `FD_READ | FD_WRITE | FD_CLOSE`.
3. Calls `WaitForMultipleObjects` on `[hEvent, s_hTerminate]` (blocking, `INFINITE` timeout).
4. On `FD_READ`: calls `OnReceived()` which buffers data and, when a complete HTTP request is detected (double `\r\n` + Content-Length body), calls `OnRequestReceived()`.
5. On `FD_WRITE`: drains the send queue chunk by chunk.
6. On `FD_CLOSE`: marks `m_bCanRecv = false`, flushes remaining send queue, exits.

### Optional HTTPS

When `thePrefs.GetWebUseHttps()` is true, mbedTLS is initialised in `StartSSL()`, and `recv`/`send` are replaced by `mbedtls_ssl_read`/`mbedtls_ssl_write`.

### Send queue (chunked)

Responses are queued as a linked list of `CChunk` nodes. If the socket send buffer is full the remaining bytes stay queued and are drained by the next `FD_WRITE` event.

```cpp
class CChunk {
    char   *m_pData;    // allocated block
    char   *m_pToSend;  // write cursor
    DWORD   m_dwSize;   // bytes remaining
    CChunk *m_pNext;
};
```

### IP whitelist

Before spawning the per-connection thread, `thePrefs.GetAllowedRemoteAccessIPs()` is checked. If the list is non-empty and the remote IP is absent, the socket is immediately closed with a log warning.

## 4. Authentication & Session Management

### Session struct

```cpp
typedef struct {
    CTime  startTime;   // creation time (used for timeout)
    long   lSession;    // session ID — rand() >> 1, 31-bit
    int    lastcat;     // last category tab viewed by this user
    bool   admin;       // true = admin, false = low-privilege guest
} Session;
```

Sessions live in `m_Params.Sessions` (an MFC `CArray<Session>`). They are lost when eMule restarts.

### Login flow

```
Client sends: GET /?w=password&p=<md5(password)>&ses=0 HTTP/1.1

1. Check bad-login table for remote IP:
   → if 4+ failures within last 15 min → HTTP 403

2. Compute MD5 of the `p=` parameter value
3. Compare against thePrefs.GetWSPass()   (admin)
            and  thePrefs.GetWSLowPass()  (guest, if enabled)

4. On match:
   a. lSession = (long)(rand() >> 1)
   b. Append Session{now, lSession, 0, isAdmin} to m_Params.Sessions
   c. Redirect to /?w=transfer&ses=<lSession>
   d. Clear bad-login entries for this IP

5. On failure:
   a. Append BadLogin{ip, now} to m_Params.badlogins
   b. Redirect to login page with error message
```

### Session validation (every request)

```cpp
_IsLoggedIn(Data, lSession)
  → iterate m_Params.Sessions
  → reset startTime to now (sliding timeout)
  → return true if found
```

### Timeout sweep

`_RemoveTimeOuts()` (alias `UpdateSessionCount()`) is called periodically. It removes sessions whose `startTime` is older than `thePrefs.GetWebTimeoutMins()` minutes.

### Guest / low-privilege

When `thePrefs.GetWSIsLowUserEnabled()` is true, a second password `GetWSLowPass()` can log in. The resulting session has `admin = false`. All write operations check `_IsSessionAdmin()` and return a "Permission denied" HTML block if the session is guest.

## 6. HTTP Behaviour & Headers

### Fixed response headers

```
Server: eMule
Connection: close
Content-Type: text/html
Content-Length: <n>
```

Gzip variant adds:
```
Content-Encoding: gzip
```

Static file responses add:
```
Last-Modified: <date>
ETag: <value>
```

### HTTP status codes in use

| Code | When issued |
|------|------------|
| `200 OK` | All dynamic pages (including application-level errors!) |
| `403 Forbidden` | Too many failed logins; file size exceeded |
| `404 Not Found` | Static file not found in webinterface/ directory |
| `500 Internal Server Error` | Memory allocation failure |

**Problem:** Application-level errors (wrong session, invalid hash, permission denied) return `200 OK` with an HTML error message inside the body. Clients cannot distinguish success from failure without parsing HTML.

### POST handling

POST body is concatenated to the URL query string with a leading `?`:

```cpp
// WebSocket.cpp
sURL = '?' + data;   // body becomes the "URL" for _ParseURL()
```

This means POST and GET parameters are treated identically — `_ParseURL()` does a simple `CString` search on whatever comes after `?`.

### Gzip compression

Applied to all dynamic pages except:
- `w=options`
- Preferences save
- HTTPS connections

Uses `zlib` deflate wrapped in a gzip header via `_GzipCompress()`.

### Character encoding

All dynamic pages emit `charset=utf-8`. URL parameters are decoded with `URLDecode(param, true)`.

### Keep-alive

`Connection: close` is always sent. Every request/response cycle creates and closes a TCP connection. There is no HTTP/1.1 keep-alive or pipelining.

## 8. Security Analysis

### 8.1 Password hashing — MD5, no salt

```cpp
// WebServer.cpp (login)
CString md5 = MD5Sum(plaintext_password).GetHash();
if (md5 == thePrefs.GetWSPass()) { /* admin */ }
```

MD5 is broken for password storage. Rainbow tables cover common passwords instantly. No salt means two users with the same password produce identical hashes.

**Fix:** Replace with Argon2id (preferred) or PBKDF2-SHA256 with a random 128-bit salt stored alongside the hash in preferences.

### 8.2 Session ID — `rand()` seeded from system clock

```cpp
lSession = (long)(rand() >> 1);
```

`rand()` is not a CSPRNG. On a predictable system or with knowledge of startup time, session IDs can be guessed.

**Fix:** Replace with `CryptGenRandom(CRYPT_EXPORTABLE, 16, bytes)` or `BCryptGenRandom` and encode as hex for the session token.

### 8.3 No thread synchronisation on shared data

Every per-request thread accesses `theApp.downloadqueue`, `theApp.uploadqueue`, and friends without holding any lock. Concurrent requests can observe torn state or trigger crashes.

**Fix:** Wrap all `theApp.*` reads in a single critical section, or move data collection into a `SendMessage`-dispatched thunk that runs on the main thread and snapshots the data safely.

### 8.4 Session stored only in process memory

Sessions are in a `CArray<Session>` with no persistence. An eMule crash or restart forces all active web users to re-authenticate.

**Fix:** Optionally persist sessions to a small SQLite file or a memory-mapped region (for same-process restart hot-reload scenarios).

### 8.5 HTTP 200 for all application errors

Errors like "wrong session", "file not found", "permission denied" all return `200 OK` with an HTML error block. Programmatic clients cannot detect failures without HTML parsing.

**Fix:** Return appropriate HTTP status codes: `401 Unauthorized` (session expired), `403 Forbidden` (insufficient privilege), `404 Not Found` (unknown hash), `422 Unprocessable Entity` (invalid parameter).

### 8.6 No CSRF protection

Forms submit via GET or POST with only a session ID in the URL. There is no CSRF token. A malicious page can silently trigger state-changing operations (cancel downloads, reboot the system) simply by loading `<img src="http://localhost:4711/?w=reboot&ses=...">`.

**Fix:** Add a per-session CSRF token to all forms; validate it on every POST that changes state.

### 8.7 Directory traversal — partially mitigated

`_ProcessFileReq` rejects paths containing `..`:

```cpp
bool filereq = sURL.GetLength() >= 3 && sURL.Find("..") < 0;
```

The check is correct for the intended static-file serving use case but relies on a single path component without full normalisation.

**Fix:** Canonicalise the path with `PathCanonicalize` and verify the resolved path is under the `webinterface/` directory before opening.

### 8.8 `w=shutdown` and `w=reboot`

Two endpoints issue OS-level shutdown/reboot commands. They are guarded by `_IsSessionAdmin`, but once an admin session is hijacked or brute-forced, an attacker can shut down the machine.

**Fix:** Disable these endpoints by default; add a separate `AllowSystemControl` preference flag; consider requiring a second password confirmation.

### 8.9 IP whitelist bypass

The IP whitelist check happens before spawning the connection thread, but is based on `sin_addr.s_addr`. On a NATted network, or when eMule is behind a proxy, the remote IP may appear as a local address, inadvertently granting access.

**Fix:** Document the limitation; consider adding CIDR-range support.

## 9. Stability & Correctness Issues

### 9.1 Terminate-thread fallback on shutdown

```cpp
// WebSocket.cpp
if (WaitForSingleObject(thread, 1300) == WAIT_TIMEOUT)
    TerminateThread(thread, 0);
```

`TerminateThread` is dangerous — it leaks resources and can corrupt heap state. The 1300 ms budget is tight for a connection that is mid-transfer.

**Fix:** Use a cooperative cancellation event (`s_hTerminate`) that request handlers check periodically.

### 9.2 `rand()` for session IDs (also a correctness issue)

`rand()` has a 15-bit to 31-bit range depending on the CRT implementation. With thousands of sessions the birthday-problem collision probability becomes non-negligible.

**Fix:** Use 128-bit cryptographic random tokens.

### 9.3 `CONNECTION: close` — no keep-alive

Every page load requires a fresh TCP handshake. For a page that references dozens of images and scripts, this causes tens of sequential round-trips. On high-latency connections the web interface becomes very slow.

**Fix:** Implement HTTP/1.1 keep-alive: track `Connection: keep-alive` requests; reset the receive buffer after each response rather than closing the socket.

### 9.4 Unbounded session array

`m_Params.Sessions` grows until `_RemoveTimeOuts()` is called (triggered from `UpdateSessionCount()` which is called on each new login). A DoS scenario: flood the login endpoint with invalid passwords — bad-login protection blocks after 4 attempts per IP, but from many IPs this populates `m_Params.badlogins` unboundedly.

**Fix:** Cap `badlogins` array at a configurable maximum; use a hash map keyed by IP with LRU eviction.

### 9.5 POST body size unchecked

The receive buffer grows dynamically:

```cpp
m_dwBufSize = dwSize + m_dwRecv + SIZE_PRESERVE;
pNewBuf = new char[m_dwBufSize];
```

A client sending an enormous `Content-Length` can exhaust process memory before the body is read.

**Fix:** Reject requests whose `Content-Length` exceeds a configurable maximum (e.g. 64 KB for normal use; larger only for file-upload endpoints if added).

### 9.6 String-replacement template engine is O(n·m)

Each page render calls `CString::Replace` for each of the ~200 tokens, scanning the entire template string each time. For a template of 115 KB with 200 tokens, this is ~23 MB of scanning per page render.

**Fix:** Pre-scan the template once at load time to build an array of (offset, token-id) pairs; render by assembling output from literal spans and token values.

## 10. Improvements Catalogue

The following are self-contained improvements, ordered from highest to lowest impact. They are designed to be applied incrementally without breaking the existing HTML interface.

---

### I-01 — Thread synchronisation (critical)

**Problem:** Race conditions between web-server threads and the main eMule thread.

**Solution:** Introduce a single `CCriticalSection g_csWebAccess`. Wrap all reads of `theApp.downloadqueue`, `theApp.uploadqueue`, `theApp.sharedfiles`, `theApp.serverlist`, `theApp.searchlist` in `CSingleLock lk(&g_csWebAccess, TRUE)`. Acquire in the data-collection phase of each `_Get*()` function, release before HTML assembly.

**Effort:** Medium. Must identify every `theApp.*` access site in `WebServer.cpp`.

---

### I-02 — Proper HTTP status codes

**Problem:** All errors return `200 OK`.

**Solution:**
- `401 Unauthorized` — session missing or expired (redirect to login)
- `403 Forbidden` — admin-only operation attempted by guest
- `404 Not Found` — unknown `filehash` or server IP
- `422 Unprocessable Entity` — missing or invalid parameter
- `500 Internal Server Error` — exception caught in `_ProcessURL`

Requires no template changes — only the HTTP response header needs to change.

**Effort:** Low.

---

### I-03 — Cryptographically secure session tokens

**Problem:** `rand()` session IDs are predictable.

**Solution:**

```cpp
// Replace
lSession = (long)(rand() >> 1);

// With
BYTE buf[16];
BCryptGenRandom(NULL, buf, 16, BCRYPT_USE_SYSTEM_PREFERRED_RNG);
// encode buf as hex string; store as CString in Session
```

Change `Session.lSession` from `long` to `CString` (or 128-bit value). Update all `lSession` comparisons.

**Effort:** Medium (session struct change touches many call sites).

---

### I-04 — CSRF tokens

**Problem:** State-changing operations are CSRF-vulnerable.

**Solution:** Generate one random 64-bit CSRF token per session. Embed it as a hidden field `<input name="csrf" value="...">` in every form. Check it on every POST operation that changes state. Reject with `403` if missing or mismatched.

**Effort:** Low–Medium.

---

### I-05 — Replace MD5 with PBKDF2-SHA256

**Problem:** MD5 password hashes without salt.

**Solution:**

```cpp
// Store: <hex-salt>:<hex-PBKDF2(password, salt, 100000, SHA256)>
// Verify: recompute and compare
```

Windows CNG provides `BCryptDeriveKeyPBKDF2`. The preferences page needs a "change password" flow that re-hashes.

**Effort:** Medium.

---

### I-06 — HTTP keep-alive

**Problem:** Every request opens a new TCP connection.

**Solution:** Parse the `Connection:` request header. If `keep-alive`, after sending the response reset the receive buffer state (`m_dwRecv = 0`) and continue the `WaitForMultipleObjects` loop rather than closing the socket. Add a per-connection idle timer (e.g. 30 s) to close stale connections.

**Effort:** Medium.

---

### I-07 — Cap POST body size

**Problem:** Unbounded request body allocation.

**Solution:** In `OnReceived()`, once `Content-Length` is parsed, reject the request with `413 Content Too Large` if it exceeds `MAX_POST_BODY` (suggest 65 536 bytes).

**Effort:** Low.

---

### I-08 — Template engine optimisation

**Problem:** O(n·m) scan per page render.

**Solution:** At template load time, scan once and build a `vector<TemplateChunk>` where each chunk is either a literal string view or a token enum. Render by iterating the chunk vector.

**Effort:** Medium-High. Requires restructuring `ReloadTemplates()` and all `_Get*()` functions.

---

### I-09 — JSON output for machine-readable endpoints

**Problem:** No structured API; HTML must be screen-scraped.

**Solution:** When `Accept: application/json` is present in the request headers, return JSON instead of HTML for all `_Get*()` functions. No template needed.

Suggested minimal JSON shapes:

**Transfers list** (`w=transfer`)
```json
{
  "downloads": [
    {
      "hash":        "A1B2C3D4...",
      "name":        "example.mkv",
      "size":        1073741824,
      "transferred": 536870912,
      "progress":    0.5,
      "state":       "downloading",
      "priority":    "normal",
      "speed":       102400,
      "sources":     42,
      "category":    0
    }
  ],
  "uploads": [
    {
      "user_name":   "SomePeer",
      "file_name":   "example.mkv",
      "transferred": 1048576,
      "speed":       20480
    }
  ]
}
```

**Server list** (`w=server`)
```json
{
  "servers": [
    {
      "ip":          "1.2.3.4",
      "port":        4661,
      "name":        "Example Server",
      "users":       100000,
      "files":       50000000,
      "ping":        48,
      "priority":    "normal",
      "static":      false,
      "connected":   true
    }
  ]
}
```

**Search results** (`w=search`)
```json

  "status":  "running",
  "results": [
    {
```

**Implementation path:** Add `Content-Type` detection in `_ProcessURL()`. If JSON mode, call a parallel `_GetTransferListJSON()` etc. that serialises directly from the same data sources.

**Effort:** High, but each endpoint can be done independently.

---

### I-10 — `CORS` headers

**Problem:** Browser-based JavaScript clients on a different origin are blocked.

**Solution:** Add `Access-Control-Allow-Origin: *` (or a configurable whitelist) to responses. Handle `OPTIONS` preflight requests.

**Effort:** Low.

---

### I-11 — Pagination for lists

**Problem:** Full download/upload/shared-file lists are assembled into one HTML page regardless of size.

**Solution:** Accept `offset=<n>&limit=<n>` parameters on `w=transfer`, `w=shared`, `w=server`. Apply them when iterating the underlying arrays. Return a total-count field in JSON mode.

**Effort:** Low–Medium.

---

### I-12 — Disable `w=shutdown` and `w=reboot` by default

**Problem:** One compromised session can reboot the machine.

**Solution:** Gate behind a new boolean preference `bWSAllowSystemControl` (default `false`). Log a warning when these endpoints are called.

**Effort:** Low.

---

### I-13 — Sanitise path in `_ProcessFileReq`

**Problem:** Single `..` check is not robust.

**Solution:**

```cpp
// Canonicalise and verify
TCHAR szCanon[MAX_PATH];
PathCanonicalize(szCanon, requestedPath);
if (_tcsncmp(szCanon, g_webInterfaceDir, g_webInterfaceDirLen) != 0)
    return HTTP 403;
```

**Effort:** Low.

---

### I-14 — Explicit HTTP verbs

**Problem:** GET requests modify state (pause, cancel, shutdown, reboot).

**Solution:** Require POST for all state-changing `op=` and `c=` operations. Return `405 Method Not Allowed` when a GET is used for a write operation. This also reinforces CSRF protection (browsers do not auto-submit cross-origin POSTs without a CORS preflight).

**Effort:** Medium (requires form method changes in both `.tmpl` files).

---

## 11. Proposed New API Layout

The following extends the existing query-string API with a parallel `/api/` prefix. The existing HTML interface is left untouched.

```
/api/v1/auth/login          POST  {password}   → {session, is_admin}
/api/v1/auth/logout         POST  —             → {}

/api/v1/transfers           GET   [cat, state, sort, offset, limit]  → {downloads[], uploads[]}
/api/v1/transfers/<hash>    GET   —              → single transfer detail
/api/v1/transfers           POST  {link}         → {hash, name}   (add ed2k link)
/api/v1/transfers/<hash>    PATCH {op, ...}      → {} (pause/resume/cancel/prio/cat/rename)
/api/v1/transfers/<hash>    DELETE —             → {} (cancel)

/api/v1/servers             GET   [sort]         → {servers[]}
/api/v1/servers             POST  {ip, port, name, priority, static}
/api/v1/servers/<ip>/<port> DELETE
/api/v1/servers/<ip>/<port>/connect    POST
/api/v1/servers/<ip>/<port>/disconnect POST

/api/v1/shared              GET   [sort, offset, limit]  → {files[]}
/api/v1/shared/<hash>/priority PATCH {priority}

/api/v1/search              POST  {query, type, method, min_mb, max_mb, ext, avail}
                                  → {job_id}
/api/v1/search/<job_id>     GET   → {status, results[]}
/api/v1/search/<job_id>     DELETE

/api/v1/kad                 GET   → {status, stats{}}
/api/v1/kad/connect         POST
/api/v1/kad/disconnect      POST

/api/v1/stats               GET   → {dl_speed, ul_speed, connections, session{}}
/api/v1/myinfo              GET   → {hash, id, tcp_port, udp_port, kad_status}

/api/v1/prefs               GET               → preferences object
/api/v1/prefs               PATCH {fields}    → updated preferences
```

All endpoints return `Content-Type: application/json`. Errors return a standard envelope:

```json
{ "error": "SESSION_EXPIRED", "message": "Session has expired, please log in again." }
```

## 12. Migration Path

The improvements above can be shipped in independent phases without breaking the existing HTML interface:

| Phase | Items | Risk |
|-------|-------|------|
| **1 — Stability** | I-01 (thread sync), I-07 (POST body cap), I-13 (path sanitise), I-12 (disable shutdown default) | Low — no API changes |
| **2 — Security** | I-02 (HTTP status codes), I-03 (CSRF tokens), I-04 (session tokens), I-05 (password hash) | Low–Medium |
| **3 — Performance** | I-06 (keep-alive), I-08 (template optimisation) | Medium |
| **4 — JSON API** | I-09 (Accept: application/json), I-10 (CORS), I-11 (pagination) | Medium–High |
| **5 — REST verbs** | I-14 (POST for writes), full `/api/v1/` routing | High |

Each phase is a self-contained PR and does not depend on the next.
