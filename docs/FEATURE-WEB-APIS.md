## 2. Architecture Overview

### Threading model

```
Main eMule UI thread
        |
        v
CWebServer (started via StartSockets / StopSockets)
        |
        |-- listening socket (CWebSocket)
        |
        +-- per-request worker thread -> _ProcessURL()
                    |
                    +-- sends WM_* Windows messages back to main UI thread
                        (WEB_GUI_INTERACTION, WEB_ADDDOWNLOADS, WEB_CATPRIO, ...)
```

Every HTTP request gets its own thread. Responses are assembled by directly reading global eMule objects (`theApp.downloadqueue`, `theApp.uploadqueue`, `theApp.sharedfiles`, `theApp.serverlist`, `theApp.searchlist`, etc.) **with no thread synchronization.**

The WebServer.cpp source contains this candid comment:
```cpp
// Here we are in real trouble! We are accessing the entire emule main thread
// data without any synchronization!! Either we use the message pump for
// m_pdlgEmule or use some hundreds of critical sections...
// For now, an exception handler should prevent the worst things.
```

### Request routing

All requests pass through a single function `_ProcessURL()`. Routing is done via the `w=` query parameter. There is no URL path routing; the "page" is always a query string value.

### Response format

Responses are **pure HTML only**, generated by string-replacement into one of two `.tmpl` files. The only structural (non-HTML) response is a small XML snippet returned for remote ed2k link additions:

```xml
<status result="OK">
  <description>...</description>
  <filename>...</filename>
</status>
```

No JSON. No content negotiation. No standard HTTP error codes for application-level errors (mostly `200 OK` everywhere).

## 3. Existing HTTP Endpoints (Full Inventory)

The server uses a **single-endpoint + query-string parameter** design. All endpoints are reached at `http://host:port/` or a path-based static file request.

### 3.1 Authentication

| URL pattern | Effect |
|-------------|--------|
| `?w=password&p=<md5hash>` | Login -- creates session, returns redirect to transfer page |
| `?w=password&p=<md5hash>&c=<ed2klink>&cat=<catid>` | Remote link addition (special combined login + add) |
| `?w=logout&ses=<sessionid>` | Destroy session |

**Session object:**
```cpp
struct Session {
    CTime  startTime;   // creation time
    long   lSession;    // random 32-bit ID (from rand())
    int    lastcat;     // last viewed category
    bool   admin;       // true = admin, false = low-privilege user
};
```

Sessions live in memory (`m_Params.Sessions` CArray) and are lost on eMule restart.
Brute-force protection: max 5 failed login attempts per IP in 15 minutes.
Optional HTTPS via mbedTLS.

### 3.2 Download / Upload Management

Reached via `?w=transfer&ses=<id>` with an `op=` action parameter.

| `op=` value | Effect |
|-------------|--------|
| `stop` | Stop a download |
| `pause` | Pause a download |
| `resume` | Resume a paused download |
| `cancel` | Cancel / remove a download |
| `getflc` | Toggle "first-last chunk" mode |
| `rename&name=<n>` | Rename download file |
| `priolow` / `prionormal` / `priohigh` / `prioauto` | Set download priority |
| `setcat&filecat=<catid>` | Assign download to a category |
| `addfriend&userhash=<h>` | Add source client as friend |
| `removefriend&userhash=<h>` | Remove from friends |

Target file identified by `file=<MD4-hash>`.

### 3.3 Server Management

`?w=server&ses=<id>&c=<command>`

| `c=` value | Parameters | Effect |
|------------|-----------|--------|
| `connect` | `ip=`, `port=` | Connect to eD2K server |
| `disconnect` | -- | Disconnect from current server |
| `remove` | `ip=`, `port=` | Remove server from list |
| `addtostatic` | `ip=`, `port=` | Mark server as static |
| `removefromstatic` | `ip=`, `port=` | Unmark static server |
| `priolow` / `prionormal` / `priohigh` | `ip=`, `port=` | Set server priority |

### 3.4 Search

`?w=search&ses=<id>`

| Parameter | Values | Effect |
|-----------|--------|--------|
| `tosearch=<query>` | string | Start search |
| `type=` | audio / video / image / document / program / archive / cdimage / emulecollection | File type filter |
| `method=` | server / global / kademlia | Search method |
| `min=<mb>&max=<mb>` | integers | File size limits in MB |
| `avail=<n>` | integer | Minimum source availability |
| `ext=<ext>` | string | File extension filter |
| `downloads[]=<hash>` | array | Add selected results to download queue |

### 3.5 Read-only Views

| `w=` value | Content |
|------------|---------|
| `transfer` | Download + upload list |
| `server` | Server list |
| `shared` | Shared files list |
| `search` | Search form + results |
| `graphs` | Traffic graphs (PNG images) |
| `log` | Application log |
| `debuglog` | Debug log |
| `sinfo` | Server information |
| `myinfo` | Client info (ID, hash, port) |
| `stats` | Statistics |
| `kad` | Kademlia DHT dialog |
| `options` | Settings page (admin only) |
| `commentlist&filehash=<h>` | File comments and ratings |

### 3.6 Settings Save (admin only)

`POST ?w=options&saveprefs=true`

Writable fields: `gzip`, `refresh`, `maxdown`, `maxup`, `maxcapdown`, `maxcapup`, `maxsources`, `maxconnections`, `maxconnectionsperfive`.

### 3.7 File Download (direct)

`?w=getfile&filehash=<MD4hash>&ses=<id>` -- Streams a shared file to the browser. Configurable max size (`MaxWebUploadFileSizeMB`).

### 3.8 System Control (admin only, if enabled in preferences)

| `w=` | Effect |
|------|--------|
| `close` | Close eMule GUI |
| `shutdown` | Shut down the OS |
| `reboot` | Reboot the OS |

## 4. Authentication Details

| Property | Current implementation |
|----------|----------------------|
| Password storage | MD5 hash (no salt) -- **cryptographically weak** |
| Session ID | `rand()` 32-bit integer -- **not cryptographically random** |
| Session lifetime | Configurable `WebTimeoutMins` |
| Session persistence | In memory only; lost on restart |
| Privilege levels | Admin / Low-privilege (read-only) |
| Brute-force guard | 5 attempts / IP / 15 min |
| Transport security | Optional HTTPS (mbedTLS) |

## 5. Architectural Constraints for REST/JSON (FEAT_020, FEAT_021)

| Constraint | Impact | Severity |
|------------|--------|----------|
| No thread synchronization on shared data | Race conditions when reading download/upload state | **CRITICAL** (FEAT_021) |
| Single monolithic `_ProcessURL()` | Hard to add new endpoint families cleanly | HIGH |
| HTML-only responses (template string replacement) | Cannot serve structured JSON | HIGH |
| No content-type negotiation (`Accept` header ignored) | Cannot selectively serve JSON vs HTML | HIGH |
| No REST verb semantics (GET modifies state) | Violates REST contract | MEDIUM |
| MD5 password, `rand()` session IDs | Weak security for exposed API | MEDIUM |
| No HTTP status codes for app errors (always 200) | Clients cannot distinguish error types | MEDIUM |
| Sessions lost on restart | Clients must re-authenticate after every eMule restart | MEDIUM |
| No pagination | Large transfer/shared-file lists loaded fully into memory | MEDIUM |
| Connection closes after every response | No keep-alive / pipelining | LOW |
| No CORS headers | Browser-based clients blocked by same-origin policy | LOW |

## 8. Implementation Roadmap

### Phase 1 -- Foundation (no visible changes to existing HTML interface)

1. **JSON response helper** -- a lightweight serializer (or pull in `nlohmann/json` as a header-only dep)
2. **Request router** -- detect `/api/v2/` path prefix and dispatch to a new handler family instead of `_ProcessURL()`
3. **Content-type negotiation** -- parse `Accept: application/json` header
4. **DTO layer** -- plain structs with `ToJson()` methods for `CPartFile`, `CUpDownClient`, `CServer`, etc.
5. **Thread safety** -- at minimum, wrap data-collection sections with `CSingleLock` on a new `CWebServerCriticalSection` (FEAT_021)

### Phase 2 -- Core read APIs (FEAT_020)

- `GET /api/v2/auth/login` + `POST /api/v2/auth/logout`
- `GET /api/v2/torrents/info` (list all transfers)
- `GET /api/v2/app/version`
- `GET /api/v2/transfer/info` (speeds + totals)
- `GET /api/v2/emule/servers`
- `GET /api/v2/emule/shared`

### Phase 3 -- Write APIs (FEAT_020)

- `POST /api/v2/torrents/add` (ed2k:// link or collection)
- `POST /api/v2/torrents/pause` / `resume` / `delete`
- `POST /api/v2/torrents/setPriority` / `setCategory`
- `POST /api/v2/emule/servers/add` / `remove`
- `POST /api/v2/app/preferences`

### Phase 4 -- Search (FEAT_020)

- `POST /api/v2/search/start` -- initiates search job, returns `{id}`
- `GET /api/v2/search/results?id=<jobid>` -- returns accumulated results + `status: running|finished`
- `POST /api/v2/search/stop`

### Phase 5 -- Polish (FEAT_020)

- Proper HTTP status codes (`400`, `401`, `403`, `404`, `409`, `500`)
- Standardised error envelope: `{"error": "HASH_NOT_FOUND", "message": "..."}`
- Pagination: `?offset=0&limit=50` on list endpoints
- CORS headers (`Access-Control-Allow-Origin`)
- Secure session IDs (replace `rand()` with `CryptGenRandom`)
- Per-session rate limiting
- Token-based auth (Bearer token as alternative to cookie session)

## 10. Summary

The existing web server is a mature but **monolithic, HTML-only, single-endpoint** system. It already covers the full feature surface of eMule -- transfers, servers, search, preferences, log, stats -- so **no new application logic needs to be written**; what is needed is a new **output layer** and **routing layer** that sit alongside the current HTML rendering code.

The biggest real risk is the **complete absence of thread synchronization** when the web server threads read download-queue and upload-queue data. Any REST API work must address this first, or race-condition crashes will be the dominant failure mode under concurrent API usage.

The cleanest approach is to add a `/api/v2/` prefix handler early in `_ProcessURL()` (or in `CWebSocket::OnRequestReceived()` before `_ProcessURL()` is called) and build the new JSON endpoints there, leaving the existing HTML path untouched.
